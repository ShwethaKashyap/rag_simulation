Chunks Created

Each paragraph has been split into a chunk with a unique ID. Chunk size affects retrieval precision — too large loses focus, too small loses context.

📌 What happens here: Each chunk is passed through an embedding model which converts text into a list of numbers (a vector). Similar meanings → similar vectors. These vectors are stored in a Vector Database, enabling fast semantic search at query time. The angle between two vectors (cosine similarity) measures how related they are — 1.0 = identical meaning, 0 = unrelated.

📌 What happens here: The user's query is embedded using the same embedding model as the documents. The query vector is compared to all stored chunk vectors using cosine similarity. The top-K most similar chunks are selected — these become the context for the LLM.

Key Concepts Summary

✂️
Chunking
Splits documents into retrievable units. Typical size: 256–1024 tokens.
🧮
Embeddings
Semantic vector representation. Models: text-embedding-3, BGE, E5.
🗄️
Vector DB
Stores & searches vectors. Examples: Pinecone, Weaviate, pgvector, ChromaDB.
📐
Cosine Similarity
Measures angle between vectors. Range: 0 (unrelated) → 1.0 (identical meaning).
🔍
Top-K Retrieval
Only the K most similar chunks are passed to the LLM to limit context size.
⚡
Prompt Augmentation
Context + query → one prompt. LLM weights unchanged. No retraining needed.
